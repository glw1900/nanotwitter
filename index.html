Technology Description

This App is built on Ruby Sinatra with ActiveRcord and a psql database. All the tests are written with MiniTest. We also adopted New Relic as our performance management tool.

We adopt redis to cache data that requires comparatively high cost in querying in database and yet are requested frequently, such as 50 most recent twitters and timeline of recent login users.

WeBrick and Puma: We use WeBrick for the server, because the Puma is actually slower in our server.

Read into comments table for every twitter is quite a cost. We noticed that most twitters might not have comments, therefore there is no need to query in the comments table every time. Then we labeled every tweet as 1 or 0, when tweets are created, the label is set to 0, then if a comment is made, we change the label to 1. Thus every time we first check the label of a twitter, if it’s labeled 0 then we check in the comment table.

Result of scalability work, timings etc.

CASE 1: 100 users, 500 tweets by test user, 30 followers. ‘/’ — maintain client load: The system crashes when the client load comes to around 2100 clients per sec. ‘/user/testuser’ — maintain client load: The system crashes when the client load comes to around 800 clients per sec. ‘/user/testuser/tweet’ — clients per second: The system crashes when the client load comes to around 1500 clients per sec.

CASE 2 : 500 users, 500 tweets by test user, 100 followers. ‘/’ — maintain client load: The system crashes when the client load comes to around 1700 clients per sec. ‘/user/testuser’ — maintain client load: The system crashes when the client load comes to around 750 clients per sec. ‘/user/testuser/tweet’ — clients per second: The system crashes when the client load comes to around 1500 clients per sec.

CASE 3 : 2000 users, 3000 tweets by test user, 1000 followers. ‘/’ — maintain client load: The system crashes when the client load comes to around 2352 clients per sec. ‘/user/testuser’ — maintain client load: The system crashes when the client load comes to around 650 clients per sec. ‘/user/testuser/tweet’ — clients per second: The system crashes when the client load comes to around 1300 clients per sec.
